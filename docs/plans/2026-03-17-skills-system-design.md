# Skills System Design

**Date**: 2026-03-17
**Status**: Approved

## Overview

Implement a progressive-disclosure skills system for the goofish-ai-customer agents, modeled after Claude Code's skills mechanism. Skills are defined as directories containing a `skill.md` file with YAML frontmatter metadata and Markdown prompt body. The system enables zero-code extensibility: adding a new skill requires only creating a directory and a `.md` file.

## Goals

- Skills are pure files (no Python required for new skills)
- Format is consistent with Claude Code skills for future reuse
- Existing `price` skill retains stateful `bargain_count` logic via declarative `state_hooks`
- LangGraph graph code requires zero changes when adding new skills

## Directory Structure

```
config/
  skills/
    price/
      skill.md          # frontmatter + prompt (with state_hooks)
    product/
      skill.md
    default/
      skill.md
  prompts/
    classify_prompt.md  # template with {skills} placeholder
    global_rules.md     # unchanged
```

Original `config/prompts/price_prompt.md`, `product_prompt.md`, `default_prompt.md` are migrated to their respective `config/skills/<name>/skill.md` files and removed.

## Skill File Format

Each skill is a directory under `config/skills/`. The main file is `skill.md`:

```markdown
---
name: price
description: 处理买家议价、砍价、询价请求
state_hooks:
  - bargain_count
---

你是一个议价助手。当前议价次数：{bargain_count}，最低价：{min_price}。

（prompt 正文）
```

### Frontmatter Fields

| Field | Required | Description |
|-------|----------|-------------|
| `name` | ✅ | Unique skill identifier; returned by classify LLM as intent value |
| `description` | ✅ | Injected into classify prompt for LLM routing decisions |
| `state_hooks` | ❌ | List of `AgentState` fields to read and inject as template variables; also drives write-back logic |

### Template Variables

- `state_hooks` fields are injected as `{field_name}` in the prompt body
- All `item_info` dict keys are also available as `{key}` template variables (e.g. `{min_price}`, `{product_name}`)
- `no_reply` is a reserved built-in intent (not a skill), routes directly to `END`

## Components

### SkillRegistry (`agents/skill_registry.py`)

Scans `config/skills/*/skill.md` at startup and manages all registered skills.

```python
@dataclass
class Skill:
    name: str
    description: str
    prompt: str              # Markdown body after frontmatter
    state_hooks: list[str]   # defaults to []
    skill_dir: Path          # path to skill directory for future file access

class SkillRegistry:
    def load_skills(self) -> dict[str, Skill]
    def get_skill(self, name: str) -> Skill
    def list_skills(self) -> list[Skill]
    def build_classify_context(self) -> str   # generates skill list for classify prompt
```

`SkillRegistry` is instantiated once in `create_agent_graph()` and injected into nodes as a dependency.

### Classify Node (updated)

`classify_prompt.md` becomes a template:

```markdown
你是一个意图分类器。根据买家消息，判断其意图类别。

可用的 skill 列表：
{skills}

仅输出以下 skill name 之一，不要输出其他内容。
```

At runtime, `{skills}` is replaced with the dynamic list generated by `registry.build_classify_context()`, e.g.:

```
- price: 处理买家议价、砍价、询价请求
- product: 处理商品详情、参数、规格咨询
- default: 打招呼、闲聊、其他通用回复
- no_reply: 无需回复（如表情、图片、系统消息）
```

### SkillExecutor Node (`agents/nodes/skill_executor.py`)

Replaces `price_node`, `product_node`, and `default_node` with a single generic executor:

1. Look up skill by `state["intent"]` from registry
2. Inject `state_hooks` fields and `item_info` keys as template variables into prompt
3. Call LLM with `[SystemMessage(prompt), *state["messages"]]`
4. Apply `check_safety()` filter
5. Write back `state_hooks` mutations (e.g. `bargain_count + 1` for price skill)
6. Return updated state

The `bargain_count` increment is driven declaratively by the presence of `bargain_count` in `state_hooks` — no skill-specific Python branching required.

### Graph Changes (`agents/graph.py`)

```
Before: classify → price / product / default → END
After:  classify → skill_executor → END
                ↘ no_reply → END
```

Conditional routing simplifies to:

```python
workflow.add_conditional_edges("classify", route_intent, {
    "no_reply": END,
    "__default__": "skill_executor",
})
workflow.add_edge("skill_executor", END)
```

**Adding a new skill = create `config/skills/<name>/skill.md`. Zero graph code changes.**

## Error Handling

| Scenario | Behavior |
|----------|----------|
| `config/skills/` directory missing | Raise error at startup, refuse to start |
| `skill.md` missing `name` or `description` | Skip skill, `logger.warning` |
| Classify returns unknown skill name | Fall back to `default` skill |
| Unresolved template variable in prompt | `logger.warning`, keep placeholder, continue |
| `bargain_count` write-back type error | Keep original value, `logger.error` |

## Testing Strategy

- **SkillRegistry unit tests**: use `tmp_path` fixture to simulate `config/skills/`, verify scan, frontmatter parsing, and `build_classify_context()`
- **SkillExecutor unit tests**: mock `SkillRegistry`, verify template variable injection and `state_hooks` write-back (especially `bargain_count + 1`)
- **Test migration**: `test_price_node.py`, `test_product_node.py`, `test_default_node.py` → migrate to test skill behavior via executor, remove standalone node tests
- **Graph integration tests**: verify end-to-end classify → skill_executor routing for all intents

## Migration Path

1. Create `config/skills/price/skill.md`, `product/skill.md`, `default/skill.md` from existing prompt files
2. Implement `SkillRegistry`
3. Update `classify_node` to use template-based prompt injection
4. Implement `skill_executor_node`
5. Update `graph.py` routing
6. Migrate and update tests
7. Delete `agents/nodes/price.py`, `product.py`, `default.py` and original prompt files
